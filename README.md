# sva-rtl-transpiler

Open-source prototype transpiler for generating synthesizable RTL checkers from a subset of SystemVerilog Assertions.

The tool has two main use cases:

1. **Hardware checker generation**: supported concurrent assertions are translated into synthesizable RTL monitors exposing assertion `pass` and `fail` outputs.

2. **Open-source formal verification support**: supported concurrent assertions are translated into explicit RTL logic plus immediate assertions, making them usable in Yosys/SymbiYosys flows where native support for concurrent SystemVerilog Assertions is limited.

The tool uses the `slang` program as a frontend and generates hardware/checker logic based on:

- a forward pipeline for antecedent evaluation;
- a checkpointed lookback engine for consequent synthesis.


## Supported features

- bounded delays `##n`, `##[n:m]`
- bounded consecutive repetition `[*n:m]`
- `throughout` conditions
- sampled-value functions such as `$past`, `$rose`, `$fell`, `$stable`, and `$changed`
- goto repetition `[->n:m]` in the antecedent
- restricted support for goto repetition `[->n:m]` as the first element of the consequent
- non-consecutive repetitions `[=n:m]` in antecedent
- outermost `or` of sequences in the antecedent
- outermost `and` of sequences in the consequent
- restricted local-variable snapshots
- `default clocking`
- `default disable iff`

## Limitations

The current implementation is a prototype and does not support the full SVA language. The main limitations are:

- **Single-clock assertions only**. The tool currently assumes one clocking event per assertion.

- **Bounded consequent sequences**. Consequent-side temporal operators must have a finite compile-time upper bound. Unbounded consequents such as `##[n:$]` or `[*n:$]` are not supported. Goto repetition `[->n:m]` is supported as the first element of the consequent.

- **Bounded failure latency**. Generated fail signals may be delayed with respect to the earliest cycle at which the original concurrent SVA property becomes semantically false. This is due to the checkpointed lookback engine used for consequent synthesis. The latency is bounded and statically determined by the generated checker structure.

- **Longer counterexamples**. When the tool is used to translate concurrent assertions into immediate assertions for Yosys/SymbiYosys, a counterexample may be longer than the shortest counterexample that a native concurrent-SVA engine could produce. The violation is still detected for the supported SVA subset, but the final immediate assertion fires according to the latency of the generated logic.

- **Limited local-variable support**. Local-variable snapshots are supported only in restricted cases where the value can be captured and propagated by the generated hardware without ambiguous temporal alignment.

- **Partial support for sequence composition operators**. Only selected forms of `or`, `and`, `throughout`, goto repetition, and non-consecutive repetition are currently supported. General compositional SVA semantics for arbitrary nested `and`, `or`, `intersect`, and `first_match` expressions is not implemented.

- **Prototype-level diagnostics**. Unsupported constructs are reported during translation, but error reporting and source-level diagnostics are limited.

## Example

A complete FIFO example is provided in `examples/fifo/`.

The example includes:

## Installation

### Prerequisites

Before installing the transpiler, make sure that `slang` is installed on your system and available in your `PATH`.
Version 10.0 or newer is required. Check that `slang` is available:

`slang --version`

If this command is not found, or the version is outdated, install slang from: `https://github.com/MikePopoloski/slang` and make sure the directory containing the `slang` executable is included in your shell `PATH`.

The transpiler requires Python 3.8 or newer.
It is recommended to update pip, setuptools, and wheel before installing the package:

`python3 -m pip install --user --upgrade pip setuptools wheel`

### Install from GitHub
Clone the repository:

`git clone https://github.com/<user>/sva-transpiler.git`

`cd sva-transpiler`

Install the transpiler:

`python3 -m pip install .`

After installation, verify that the command is available:

`sva_transpiler --help`

## Quick start
The transpiler does not parse SystemVerilog directly. It expects a JSON AST generated by `slang`.
Assume that the file to be transpiled is `design.sv`

```
`define DATA_SIZE 16
`define FIFO_LOCATIONS 16

module test #(parameter N = `DATA_SIZE, K = `FIFO_LOCATIONS)
 (input  logic clk, rst, wr_en, rd_en,
  input  logic [N-1:0] wr_data,
  input  logic full, empty,
  input  logic [N-1:0] rd_data,
  input logic [$clog2(K)-1:0] wr_ptr, rd_ptr
);

`define WR_FIRE (wr_en && !full)
`define RD_FIRE (rd_en && !empty)

default clocking @(posedge clk); endclocking
default disable iff (rst);

property empty_to_full; 
  empty ##0 ((!`RD_FIRE) throughout (`WR_FIRE[->`FIFO_LOCATIONS]))
      |=>  full;
endproperty

a1: assert property (empty_to_full);

endmodule
```

First, generate the AST with source-location information:

`slang --ast-json-source-info  --ast-json  design.json  design.sv`

For larger designs, pass to `slang` the same include directories, defines, source files, and top-module options normally required to compile the design.

Then, run the transpiler:

`sva_transpiler design.json  -o design_transpiled.sv`

The following should be produced:

```
Info: Source file identified: design.sv
Assertion 1:
  - Label: a1
  - Type: assert
  - Clock: posedge clk (default clocking)
  - Reset: rst (default disable)
  - Local Vars: NONE
  - Source Line: 23


========================================
          TRANSPILER SUMMARY          
========================================
 Assertions translated : 1
 Covers translated     : 0
 Assumptions translated: 0
----------------------------------------
 ERRORS (Skipped)      : 0
----------------------------------------
 TOTAL GENERATED       : 1
========================================
```

The output file contains the original assertion commented out and generated checker logic inserted into the module.

## Default Mode

Default mode is useful for open-source formal flows based on Yosys/SymbiYosys, where native support for concurrent SystemVerilog Assertions is limited. The transpiler converts supported concurrent assertions into RTL code, followed by immediate `assert`, `assume`, or `cover` statements.

The generated immediate assertions are not a textual one-to-one replacement of the original concurrent assertions. Temporal behavior is implemented by the generated checker logic. As a consequence, the cycle in which a generated immediate assertion fails may be later than the earliest cycle at which the original concurrent SVA property becomes semantically falsified.

Therefore, when using the generated immediate assertions, a counterexample may be longer than the shortest counterexample that a native concurrent-SVA engine could produce. The violation is still detected for the supported bounded subset, but the reported failure point follows the latency of the generated checker logic.

In default mode, the transpiler does not modify the module interface. The original supported concurrent assertions are commented out and replaced by generated SystemVerilog checker logic inside the original module. The generated logic evaluates the translated property and emits final immediate `assert`, `assume`, or `cover` statements.

Example:

For an assertion such as:

`my_assert: assert property (...);`

the transpiler generates internal checker logic and a final immediate assertion associated with `my_assert`.

The behavior of generated immediate assertions can be customized with:

`--assert-action none`
`--assert-action display`
`--assert-action error`

For cover properties, the generated cover behavior can be customized with:

`--cover-action none`
`--cover-action display`

## Checker Mode

In checker mode, the transpiler exposes assertion result signals as module outputs. This mode is intended for hardware-checker generation and flows where assertion results must be visible as RTL outputs.

The generated `*_assert_fail` output may also be delayed with respect to the earliest semantic failure point of the original SVA property. This latency is bounded for supported SVA and is determined by the generated checkpoint schedule.

For an assertion labeled `<label>`, the generated fail output is named:

`<label>_assert_fail`

This signal pulses when the assertion is violated, with a possible latency as described above.

To select checker mode, use `--checker`:

`sva_transpiler --checker design.json -o design_transpiled.sv`

To enable also pass-signal generation use `--pass`:

`sva_transpiler --checker --pass design.json -o design_transpiled.sv`

When pass-signal generation is enabled, the transpiler also generates:

`<label>_assert_pass`

This signal pulses when the assertion is verified in a non-vacuous way.

In checker mode, options  `--assert-action`, `--cover-action`, and `--ifdef-mode` are not supported.
