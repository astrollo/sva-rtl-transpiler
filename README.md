# sva-rtl-transpiler

Open-source prototype transpiler for generating synthesizable RTL checkers from a subset of SystemVerilog Assertions.

The tool uses the `slang` program as a frontend and generates hardware checkers based on:
- a forward pipeline for antecedent evaluation;
- a checkpointed lookback engine for consequent synthesis.

## Supported features

- bounded delays `##n`, `##[n:m]`
- bounded consecutive repetition `[*n:m]`
- `throughout` conditions
- sampled-value functions such as `$past`, `$rose`, `$fell`, `$stable`, and `$changed`
- goto repetition `[->n:m]` in the
- goto repetition `[->n:m]` as the first element of the consequent
- non-consecutive repetitions `[=n:m]` in antecedent
- outermost `or` of sequences in the antecedent
- outermost `and` of sequences in the consequent
- restricted local-variable snapshots
- `default clocking`
- `default disable iff`

## Limitations
(too many to be listed extensively...)
- single-clock assertions
- bounded consequent sequences
- limited support for local variables

## Installation

### Prerequisites

Before installing the transpiler, make sure that `slang` is installed on your system and available in your `PATH`.
Version 10.0 or newer is required. Check that `slang` is available:

`slang --version`

If this command is not found, or the version is outdated, install slang from: `https://github.com/MikePopoloski/slang` and make sure the directory containing the `slang` executable is included in your shell `PATH`.

The transpiler requires Python 3.8 or newer.
It is recommended to update pip, setuptools, and wheel before installing the package:

`python3 -m pip install --user --upgrade pip setuptools wheel`

### Install from GitHub
Clone the repository:

`git clone https://github.com/<user>/sva-transpiler.git`
`cd sva-transpiler`

Install the transpiler:

`python3 -m pip install .`

After installation, verify that the command is available:

`sva_transpiler --help`

## Basic Usage
The transpiler does not parse SystemVerilog directly. It expects a JSON AST generated by `slang`.
Assume that the file to be transpiled is `design.sv`

```
`define DATA_SIZE 16
`define FIFO_LOCATIONS 16

module test #(parameter N = `DATA_SIZE, K = `FIFO_LOCATIONS)
 (input  logic clk, rst, wr_en, rd_en,
  input  logic [N-1:0] wr_data,
  input  logic full, empty,
  input  logic [N-1:0] rd_data,
  input logic [$clog2(K)-1:0] wr_ptr, rd_ptr
);

`define WR_FIRE (wr_en && !full)
`define RD_FIRE (rd_en && !empty)

default clocking @(posedge clk); endclocking
default disable iff (rst);

property empty_to_full; 
  empty ##0 ((!`RD_FIRE) throughout (`WR_FIRE[->`FIFO_LOCATIONS]))
      |=>  full;
endproperty

a1: assert property (empty_to_full);

endmodule
```

First, generate the AST with source-location information:

`slang --ast-json-source-info  -a-st-json  design.json  design.sv`

For larger designs, pass to `slang` the same include directories, defines, source files, and top-module options normally required to compile the design.

Then, run the transpiler:

`sva_transpiler design.json  design_transpiled.sv`

The following should be produced:

```
Info: Source file identified: design.sv
Assertion 1:
  - Label: a1
  - Type: assert
  - Clock: posedge clk (default clocking)
  - Reset: rst (default disable)
  - Local Vars: NONE
  - Source Line: 23


========================================
          TRANSPILER SUMMARY          
========================================
 Assertions translated : 1
 Covers translated     : 0
 Assumptions translated: 0
----------------------------------------
 ERRORS (Skipped)      : 0
----------------------------------------
 TOTAL GENERATED       : 1
========================================
```

The output file contains the original assertion commented out and generated checker logic inserted into the module.

## Default Mode
By default, the transpiler does not modify the module interface.

In this mode, the original supported concurrent assertions are commented out and replaced by generated SystemVerilog checker logic inside the original module. The generated logic evaluates the translated property and emits final immediate `assert`, `assume`, or `cover` statements.

Default mode is intended for simulation-oriented or formal-oriented flows with yosys/sby where immediate assertions are acceptable.

Example:

For an assertion such as:

`my_assert: assert property (...);`

the transpiler generates internal checker logic and a final immediate assertion associated with `my_assert`.

The behavior of generated immediate assertions can be customized with:

`--assert-action none`
`--assert-action display`
`--assert-action error`

For cover properties, the generated cover behavior can be customized with:

`--cover-action none`
`--cover-action display`

## Checker Mode

In checker mode, the transpiler exposes assertion result signals as module outputs. This mode is intended for hardware-checker generation and flows where assertion results must be visible as RTL outputs.

For an assertion labeled `<label>`, the generated fail output is named:

`<label>_assert_fail`

This signal pulses when the assertion is violated.

To select checker mode, use `--checker`:

`sva_transpiler --checker design.json -o design_transpiled.sv`

When pass-signal generation is enabled, the transpiler also generates:

`<label>_assert_pass`

This signal pulses when the assertion is verified in a non-vacuous way.

To enable pass-signal generation:

`sva_transpiler --checker --pass design.json -o design_transpiled.sv`

In checker mode, options  `--assert-action`, `--cover-action`, and `--ifdef-mode` are not supported.
